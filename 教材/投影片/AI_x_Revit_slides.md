# 📋 課程協作方式

**本課程採用 Git 協作**

```
┌──────────────────┐
│  老師的 Repo     │ ← 您正在看的原始版本
└────────┬─────────┘
         │ Fork（複製到您的帳號）
         ▼
┌──────────────────┐
│  您的 Repo       │ ← 屬於您自己的版本
└────────┬─────────┘
         │ Clone（下載到電腦）
         ▼
┌──────────────────┐
│  您的電腦        │ ← 實際操作的地方
└──────────────────┘
```

<!-- note:
在開始課程前，我們先確認大家都用正確的方式取得教材。

這邊要先講 Fork 這個概念。你可以把它想成是建築業常見的那種情境：甲方給了一份設計準則，你不會直接在甲方的檔案上面改，對吧？你會先複製一份到自己的資料夾，然後在自己的版本上面作業。這樣你改壞了也不會影響到原版，而且你做出好東西的時候，還可以回頭跟甲方說「欸我這邊有個不錯的做法，要不要參考看看」。

Fork 就是這個意思。你把我的專案複製一份到你自己的 GitHub 帳號底下，然後在你自己的版本上面玩。你開發出好用的工作流程，可以發 Pull Request 貢獻回來，我審核通過之後，其他同學也能用。

所以現在請確認一下：你的 GitHub 帳號底下有沒有這個專案？網址是不是 github.com/你的帳號/REVIT_MCP_study？如果不是的話，現在就去 Fork 一下。

# 🔄 知識貢獻流程

**當您開發出好用的 domain：**

| 步驟 | 指令 | 說明 |
|:----|:----|:----|
| 1️⃣ 產生 | `/domain ...` | 讓 AI 幫你產生 SOP |
| 2️⃣ 提交 | `git add domain/新檔.md` | 加入版本控制 |
| 3️⃣ 推送 | `git commit + push` | 上傳到您的 Repo |
| 4️⃣ 貢獻 | 發 Pull Request | 請老師審核合併 |

<!-- note:
當你用 AI 開發出一個好用的工作流程，你可以分享給全班。

這個流程其實跟事務所裡面的知識管理很像。你今天做了一個案子，發現某個檢討流程特別順，你會不會想把它記下來，讓下次做類似案子的同事可以參考？在傳統事務所，這種東西通常是變成一份 Word 檔，然後丟到公用資料夾的某個角落，三個月後沒人記得它在哪裡。

這邊我們用 Git 來做這件事。你用 AI 跑完一個成功的流程之後，打 /domain 讓 AI 幫你把它存成一份 SOP 文件。這份文件會存在 domain 資料夾底下。然後你用 git add、git commit、git push 把它推上去，再到 GitHub 發一個 Pull Request。我審核過了之後，這個流程就變成大家都能用的公共資產了。

對了，只提交 domain/ 和 GEMINI.md 的變更就好，其他程式碼檔案不要動。不然我 review 起來會很痛苦。

# 🔃 同步老師的更新

**定期取得其他人的貢獻：**

1. 到 GitHub 您的 Repo 頁面
2. 點擊「Sync fork」按鈕
3. 本機執行 `git pull`

```bash
# 取得最新內容
git pull
```

<!-- note:
當我合併了其他同學的貢獻之後，你需要把這些更新同步過來。

這個步驟很簡單。你進到你自己的 GitHub Repo 頁面，會看到一行字寫「This branch is X commits behind」，意思是你的版本比我的版本落後了幾個版本。這時候你點「Sync fork」然後按「Update branch」就可以了。

同步完 GitHub 之後，記得回到你電腦上的專案資料夾，執行 git pull 把最新的內容拉下來。

我建議每週上課前同步一次。不然隔太久，你的版本跟我的版本差太多，到時候要 merge 會比較麻煩。就像工地施工圖如果太久沒有跟設計變更同步，到時候圖紙打架會很慘的那樣。

# 技術架構簡化版

![bg right:50%](./images/unified_03_architecture_1767752119600.png)

**三層結構**：
- 你 = 業主/建築師
- AI = 總工程師
- Revit = 工地現場

你說話 → AI 翻譯 → Revit 執行

<!-- note:
這張圖是你今天唯一需要理解的技術概念。其他那些什麼 WebSocket、什麼 MCP Server，你可以當作不存在，不影響你使用這套系統。

把這個系統想像成一個建築專案的分工架構。最上層是你，就是業主加建築師的角色，負責發出需求：「我想要一棟防火性能好的房子」。中間層是 AI，它的角色就像總工程師，負責把你的需求翻譯成具體的施工步驟。最下層是 Revit，它就是工地現場，負責實際去執行那些操作。

你說：「把防火時效不足的牆標紅色」。AI 聽懂之後，把這句話拆解成一連串 Revit 可以理解的指令——先查詢所有牆體、讀取防火時效參數、篩選出不合規的、對這些牆套用紅色覆寫。然後 Revit 就照著執行，牆就變紅了。

你不需要知道中間到底怎麼運作的，就像你打電話不需要知道訊號是怎麼傳到基地台的。你只要會說話就可以了。

# ⭐ 自然語言 vs 斜線指令

![bg right:50%](./images/unified_04_nlp_vs_command_1767752151052.png)

| 特性 | 自然語言 | 斜線指令 |
|:----|:-------:|:-------:|
| 彈性 | 高 | 低 |
| 一致性 | 不定 | 相同 |
| 適合 | 探索新任務 | 重複執行 |

**第一次用自然語言，成功後變斜線指令**

<!-- note:
這張圖是今天最重要的概念，沒有之一。

左邊是自然語言對話——就像我們平常聊天，想到什麼說什麼。「欸幫我看一下這案子的防火檢討有沒有問題」「走廊好像太窄了幫我確認一下」，這種說法彈性很大，AI 會盡量理解你的意思然後去做。

右邊是斜線指令——像工地的標準作業程序，寫好了就照著走。每次執行的步驟都一樣，結果也比較一致。

使用時機是這樣的：第一次做一件事情，你不確定怎麼做最好，用自然語言去探索。等你找到一個成功的做法了，就把它變成斜線指令，下次直接用。

舉個例子，你今天第一次做防火檢查，可能要跟 AI 來回對話好幾輪才找到正確的流程。但做完之後，你用 /domain 把這個流程記錄下來。下次再做類似案子的時候，直接叫出那個流程就好，不用從頭聊起。

這個概念很像工地的 SOP。工班師傅第一次做某個工項，可能需要現場討論、試錯。但做過一次之後，就有經驗了，下次照著做就好。我們現在要學的就是怎麼把這個「經驗」記錄下來讓 AI 可以重複使用。
-->

# 知識累積流程

![bg right:50%](./images/unified_12_knowledge_mgmt_1767752335259.png)

```
成功對話
    ↓
┌───────────┬───────────┐
│ /lessons  │ /domain   │
│ 高階規則  │ SOP流程   │
│ →GEMINI.md│ →domain/  │
└───────────┴───────────┘
```

<!-- note:
這張圖解釋了兩種知識累積的方式。

場景是這樣的：你剛剛跟 AI 合作完成了一次走廊寬度檢查，整個流程很順，結果也正確。接下來你有兩個選擇。

第一個選擇是用 /lessons。這個指令是用來記錄「規則」和「經驗教訓」的。比方說你發現走廊識別的時候，不能只認「走廊」這兩個字，因為有些案子會叫「廊道」，有些會叫「Corridor」，日本案子可能是「廊下」。這種多語言容錯的知識就適合用 /lessons 記下來，AI 會把它寫進 GEMINI.md 這個檔案裡面。

第二個選擇是用 /domain。這個指令是用來記錄「完整流程」的。比方說這次走廊檢查從頭到尾怎麼做、用了哪些工具、參數怎麼設定，全部都記錄下來變成一份 SOP 文件，存在 domain 資料夾底下。下次有人說「幫我做走廊檢查」，AI 就會自動載入這份文件，照著流程執行。

簡單講，/lessons 是記「心法」，/domain 是記「招式」。心法讓 AI 變聰明，招式讓 AI 可以直接幹活。

# 二、MCP 架構解析
## 用建築視角理解系統

![bg right:50%](./images/unified_03_architecture_1767752119600.png)

```
你 + AI Client
    ↓ 自然語言
MCP Server（翻譯與調度）
    ↓ WebSocket:8964
Revit Add-in（現場執行）
```

<!-- note:
上一堂課我們看過這張圖，今天我們要更深入地理解每一層在做什麼。

把這個系統想像成一個建築專案的組織架構。最上層是你跟 AI Client，這就像是業主加建築師的組合——業主說「我要一棟安全的房子」，建築師把這個需求具體化成設計圖。中間層是 MCP Server，它的角色就像總工程師辦公室，負責把設計圖翻譯成施工計畫，然後分派給各個專業分包。最下層是 Revit Add-in，這就是工地現場，真正在動手做事的地方。

中間那個 WebSocket:8964 是什麼？你可以把它想成工地對講機。總工程師辦公室跟工地現場之間需要即時通訊，有什麼狀況馬上講、馬上回報，這就是 WebSocket 在做的事情。8964 是頻道號碼，就像對講機要調到同一個頻道才能通話。

你不需要記住這些技術名詞。你只要知道：你說話，AI 翻譯，Revit 執行。這樣就夠了。

# 六種檔案類型（上）

![bg right:55%](./images/unified_05_file_types_1767752172640.png)

| 檔案 | 建築類比 |
|:----|:--------|
| **.json** | 聯絡簿、設定表 |
| **.ts/.js** | 總工程師手冊 |
| **.cs** | 施工手冊 |

<!-- note:
這邊要介紹你會在這個專案裡面看到的幾種檔案類型。不用怕，你不需要會寫這些東西，但至少要知道它們是什麼，這樣出問題的時候才知道要去哪裡找。

第一種是 .json 檔案，這是設定檔。你可以把它想成專案的聯絡人清單跟設定表。上面寫著：MCP Server 的位置在哪裡、要連接哪個 port、有哪些工具可以用。就像工程開工前會有一份設定表，寫著專案經理電話、工地地址、開工日期這些資訊。

第二種是 .ts 跟 .js 檔案，這是 TypeScript 跟 JavaScript，是 MCP Server 的程式碼。你可以把它想成總工程師的工作手冊，上面寫著：「如果收到『建牆』的需求，要先確認長度、材料、位置，然後發派給工地」。這些邏輯都寫在這裡面。

第三種是 .cs 檔案，這是 C# 程式碼，是 Revit Add-in 的原始碼。你可以把它想成工地師傅的施工手冊，上面寫著：「如果接到 create_wall 的指令，就拿起工具，照著這個步驟砌一面牆」。很具體、很細節的操作都在這裡。

# 六種檔案類型（下）

![bg right:55%](./images/file_types_part2_1767766799295.png)

| 檔案 | 建築類比 |
|:----|:--------|
| **.dll** | 預鑄構件（編譯成品） |
| **.addin** | 進場許可證 |
| **.md** | SOP 規範手冊 |

<!-- note:
接下來是另外三種檔案類型。

第四種是 .dll 檔案，這是編譯後的程式庫。你可以把它想成預鑄構件。.cs 是設計圖，是原始碼；.dll 是工廠生產出來的成品，是可以直接拿去現場裝的東西。如果你改了 .cs 的內容，要記得「重新生產」一次，也就是重新編譯，才會產生新的 .dll。

第五種是 .addin 檔案，這是 Revit 的外掛設定檔。你可以把它想成進場許可證。Revit 是一個管得很嚴的工地，外人想進去要先有許可證。這個 .addin 檔案就是告訴 Revit：「這個外掛是合法的，讓它進來」。沒有這張許可證，Revit 根本不會理你的外掛。

第六種是 .md 檔案，這是 Markdown，就是你現在看到的這些文件格式。你可以把它想成 SOP 規範手冊，是寫給 AI 看的。上面寫著：「如果有人說『檢查防火』，就照著這些步驟做」。這就是我們的 domain/*.md 檔案在做的事情。

# 安裝四步驟

![bg right:50%](./images/unified_06_installation_1767752194897.png)

1. **下載** Node.js
2. **編譯** .cs → .dll
3. **部署** 到 Revit Addins 目錄
4. **連接** AI Client 設定

預計時間：25 分鐘

<!-- note:
這張圖是安裝流程的總覽，從左到右依序是四個步驟。

第一步是下載 Node.js。Node.js 是 MCP Server 運行的環境，就像你要跑 Revit 需要先裝 Windows 一樣，你要跑 MCP Server 需要先裝 Node.js。

第二步是編譯。把 .cs 原始碼變成 .dll 可執行檔。這就像把設計圖送去工廠生產成預鑄構件。

第三步是部署。把編譯好的 .dll 跟 .addin 檔案放到 Revit 指定的資料夾底下。這就像把預鑄構件運到工地，放到正確的位置。

第四步是連接。設定你的 AI Client，讓它知道要去哪裡找 MCP Server。這就像把對講機調到正確的頻道。

整個流程大概 25 分鐘可以完成。等一下我們會一步一步帶著做。
-->

# ⚠️ 常見問題排解

![bg right:45%](./images/unified_14_troubleshoot_1767752381900.png)

1. **PowerShell 權限錯誤**
   `Set-ExecutionPolicy RemoteSigned`

2. **Port 8964 被佔用**
   `netstat -ano | findstr :8964`

3. **DLL 路徑錯誤**
   確認使用正確的 .csproj

<!-- note:
安裝過程中最常遇到的問題有三個，我先講一下怎麼解決。

第一個是 PowerShell 權限錯誤。Windows 預設不讓你執行腳本，這是安全機制。解法是打開 PowerShell，輸入 Set-ExecutionPolicy RemoteSigned，然後確認。這就像是你要進工地，警衛擋著不讓進，你出示工作證就可以了。

第二個是 Port 8964 被佔用。如果你之前開過 Revit MCP 沒有正常關閉，那個 port 可能還被佔著。解法是用 netstat 指令找出是誰在佔用，然後把它關掉。這就像工地的某個區域被之前的工班佔著沒清場，你要先請他們離開才能進場。

第三個是 DLL 複製到錯誤的位置。2022/2023 跟 2024 的輸出路徑不一樣，要確認你用的是正確的 .csproj 檔案。這就像預鑄構件送到工地，結果放錯樓層，當然裝不起來。

# 案例四：元素上色

![bg right:55%](./images/unified_10_coloring_1767752294318.png)

**標準 5 步驟流程**

為什麼需要 5 步驟？
→ 牆柱接合時上色會出問題

<!-- note:
最後一個案例是元素上色。

上色這件事看起來很簡單，不就是選元素、改顏色嗎？但在 Revit 裡面有一個陷阱：如果牆跟柱子有接合關係，上色的時候顏色可能會「漏」到別的元素上面，或是根本上不到你要上的那個元素。

為什麼會這樣？因為 Revit 的接合機制會讓兩個元素的幾何合併在一起，視覺上變成一體。這時候你對其中一個上色，可能會影響到另一個。

所以我們有一個標準的五步驟流程，專門用來處理這個問題。等一下我會一步一步帶著做。
-->

---

# 上色 5 步驟

![bg right:40%](./images/coloring_steps_1767766825505.png)

```
1. 清除舊顏色
       ↓
2. 取消牆柱接合 ← 重要！
       ↓
3. 牆體上色
       ↓
4. 柱子上色（黑色）
       ↓
5. 恢復牆柱接合
```

<!-- note:
上色的五步驟是這樣的。

第一步，清除舊顏色。如果這個視圖之前有做過顏色覆寫，先全部清掉，從乾淨的狀態開始。

第二步，取消牆柱接合。這一步很關鍵。請 AI 把當前視圖裡面所有牆跟柱子的接合關係解除。這樣牆就是牆、柱子就是柱子，不會互相影響。

第三步，對牆體上色。這時候因為接合已經解除了，你可以放心地根據防火時效或其他屬性來分色，顏色會正確地套用在每面牆上面。

第四步，對柱子上色。通常我們會把柱子統一標成黑色或深灰色，讓它們跟牆區分開來。

第五步，恢復牆柱接合。上色完成之後，把接合關係恢復原狀。這樣模型的幾何還是正確的，只是視覺上多了顏色標記。
-->

# 協作開發流程

![bg right:50%](./images/unified_11_tool_dev_1767752312915.png)

```
1. 你：說明需求
2. AI：查詢 API、設計邏輯
3. AI：產生程式碼
4. 你：測試、回報結果
5. 重複 3-4 直到完成
```

<!-- note:
工具開發的流程是這樣的，一個來回迭代的過程。

第一步，你說明需求。盡量講清楚你要做什麼、輸入是什麼、預期輸出是什麼。比方說：「我需要一個工具，輸入樓層名稱，輸出那層樓所有房間的面積總和」。

第二步，AI 去查 Revit API，確認技術上可行，然後設計邏輯。AI 會告訴你它打算怎麼做，比方說：「先用 get_rooms_by_level 取得房間清單，然後加總面積欄位」。

第三步，AI 產生程式碼。如果是簡單的功能，可能直接用現有工具組合就可以。如果需要新的 API 呼叫，AI 會產生 C# 或 TypeScript 的程式碼。

第四步，你去測試。把程式碼加進專案、重新編譯、跑一次看看結果對不對。如果有問題，把錯誤訊息或不正確的結果回報給 AI。

第五步，重複三跟四，直到功能完成。這個過程可能要來回幾次，特別是比較複雜的功能。但每一次迭代，AI 都會根據你的回饋修正，越來越接近你要的結果。
-->

# /lessons vs /domain

![bg right:40%](./images/lessons_vs_domain_1767766878851.png)

| | /lessons | /domain |
|:----|:---------|:--------|
| **用途** | 記規則 | 記流程 |
| **儲存** | GEMINI.md | domain/*.md |
| **觸發** | 不自動 | 關鍵字觸發 |

**同一個成功案例可以兩種都記**

<!-- note:
這個對比很重要，我們再講一次。

/lessons 是用來記「規則」的。這些規則會被寫進 GEMINI.md，成為 AI 的背景知識。比方說「走廊識別要支援多語言」這種經驗教訓，就適合用 /lessons 記。AI 不會自動觸發這些規則，但在處理相關問題的時候會參考。

這兩個指令不是互斥的。同一個成功案例，你可以兩種都記。用 /lessons 記下你學到的經驗教訓，用 /domain 記下完整的操作流程。這樣 AI 既有心法、又有招式。

# Domain 管理技巧

**當 domain/ 太多時怎麼辦？**

```
/review 請檢查 domain/ 目錄，
找出可能重複或過時的檔案
```

AI 會回報：
- 相似功能的檔案（建議合併）
- 過時的流程（建議更新/刪除）
- 缺少測試案例的檔案

<!-- note:
隨著你越用越多，domain 資料夾裡面的檔案會越來越多。這時候會遇到一個問題：我怎麼知道有哪些 domain 可以用？會不會有重複的？會不會有些已經過時了？

這時候可以用 /review 指令。你跟 AI 說：「請檢查 domain 目錄，找出可能重複或過時的檔案」。AI 會掃描整個資料夾，分析每個檔案的內容，然後告訴你：

「corridor-analysis.md 和 escape-route.md 功能相似，建議合併。」
「fire-rating-check-v1.md 已經有 v2 版本了，舊版建議刪除。」
「floor-area-review.md 缺少測試案例，建議補充。」

這就像辦公室定期整理檔案櫃一樣。每隔一段時間做一次 review，把重複的合併、過時的刪除、不完整的補齊。這樣你的知識庫才會保持乾淨、好用。

# Skill 概念

![bg right:40%](./images/unified_13_future_skill_1767752364103.png)

**Skill = 把多個 Tools 包裝成可重複使用的模組**

```
Skill（技能包）
├── Tool 1：get_rooms_by_level
├── Tool 2：override_graphics
├── Tool 3：check_regulation
└── 觸發邏輯 + 測試案例
```

**Tools** = 我們一直在用的 MCP 工具
**Skill** = 包裝好的自動化流程

<!-- note:
Skill 是什麼？你可以把它想成「技能包」。

我們一直在用的那些功能，像是 get_rooms_by_level、override_graphics、query_elements，這些都是單一的 Tool，每個 Tool 做一件事情。

Skill 是把多個 Tools 包裝在一起，加上觸發邏輯跟測試案例，變成一個完整的自動化流程。

比方說「容積檢討」這個 Skill，裡面包含：取得房間清單、分類計入不計入、計算面積、對照法規文件。這些步驟串起來，就是一個完整的技能包。你只要說「執行容積檢討」，整個流程就會自動跑完。

目前我們用 domain 文件來達成類似的效果。未來 MCP 可能會有更正式的 Skill 格式，讓這種打包更系統化、更好管理。

# 現在可以做的準備

![bg right:40%](./images/preparation_checklist_1767766868435.png)

2. ✅ 為每個 domain 補充範例對話

雖然正式的 Skill 格式還沒出來，但你現在就可以開始為未來做準備。

<!-- note:
在結束之前，回答幾個常見問題。

第一個問題：AI 會取代建築師嗎？

不會。AI 做的是重複性、機械性的工作——查資料、對法規、算面積、上顏色。這些事情它做得比人快、比人準。但設計判斷、創意發想、跟業主溝通、處理模糊需求，這些還是要靠人。AI 是輔助工具，不是替代品。

第二個問題：需要付費嗎？

視情況而定。AI 平台有些是免費的、有些要付費、有些是用量計價。這門課用的工具本身是開源的，不用錢。但如果你用 Claude 或 Gemini 這些商業平台，超過免費額度可能就要付費。具體方案建議直接去各平台網站確認。

第三個問題：公司的模型能上傳到 AI 嗎？有沒有資安疑慮？

模型資料是留在你電腦上的，不會被上傳到雲端。傳給 AI 的只有操作指令跟參數值，不包含完整的模型檔案。當然，每個 AI 平台的隱私政策不一樣，如果公司有嚴格的資安要求，建議跟 IT 部門確認。
