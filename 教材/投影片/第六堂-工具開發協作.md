## Slide 2: 評估現有工具能力

**標題**：先問 AI：能不能做到？

---

### 📝 講師備忘錄

**評估流程（10 分鐘）**：

```
你的需求
    ↓
問 AI：「根據現有工具，能不能完成 XXX？」
    ↓
   ┌─────────┐
   │  能完成 │ → AI 組合工具執行
   └────┬────┘
        │
   ┌────▼────┐
   │ 不能完成 │ → 需要新增工具
   └─────────┘
```

---

**提問範本**（帶學員一起練習）：

```
你：請根據以下現有工具清單，評估能否完成「自動標記消防栓位置」：
- query_elements
- get_element_info
- override_element_graphics
- create_dimension

如果可以，請說明如何組合這些工具。
如果不行，請說明缺少什麼能力。
```

---

**AI 可能的回應**：

```
分析結果：
✅ 可以找到消防栓（用 query_elements）
✅ 可以上色標記（用 override_element_graphics）
❌ 無法自動放置標註文字（缺少 place_annotation_symbol 工具）

建議：需要開發新工具來放置標註。
```

## Slide 3: Revit SDK 探索

![Revit SDK 圖示](./images/unified_15_revit_sdk_1767752398709.png)

**標題**：Revit API 文件是說明書

---

### 📝 講師備忘錄

**SDK 介紹（10 分鐘）**：

> 「這張圖顯示：一本打開的書（文件）、程式碼圖示、建築模型。
> 
> 那本書就是 Revit SDK 文件——Revit 的『完整使用說明書』。
> 
> 你不需要自己讀它，你只需要告訴 AI：『幫我查查看有沒有這個功能。』」

---

**Revit SDK 網站**：
- https://www.revitapidocs.com/

**版本差異提醒**：
- 2022/2023：ElementId 用 `int`
- 2024：ElementId 用 `long`

> 「這些技術細節你不用記，AI 會處理。但你要知道『有版本差異』這件事。」

---

**對話範例**：

```
你：我想做一個工具，可以自動把門的寬度寫在門旁邊。
請查詢 Revit API 有沒有這個功能，並告訴我需要用什麼 API。

AI：根據 Revit API 文件，你需要使用：
1. FamilyInstance 類別取得門的資訊
2. IndependentTag.Create() 方法建立標註
3. 需要指定 TagMode 和 TagOrientation...
```

## Slide 4: Prompt 開發流程

**標題**：五步驟工具開發

---

### 📝 講師備忘錄

**帶學員走過完整流程（20 分鐘）**：

---

**步驟 1：描述需求**
```
你：我需要一個工具，可以自動在每個房間的中心放置面積標籤
```

---

**步驟 2：請 AI 查詢 SDK**
```
你：請查詢 Revit API Documentation，找出可以實現這個功能的 API
```

---

**步驟 3：請 AI 設計規格**
```
你：請根據 Revit SDK 的 RoomTag 類別，設計一個 MCP 工具的規格
```

AI 回應：
```typescript
{
  name: "place_room_tag",
  description: "在房間中心放置面積標籤",
  inputSchema: {
    properties: {
      roomId: { type: "number", description: "房間 ID" },
      tagTypeId: { type: "number", description: "標籤類型 ID" }
    }
  }
}
```

---

**步驟 4：請 AI 產生程式碼**
```
你：請產生 TypeScript 工具定義和 C# 實作
```

---

**步驟 5：測試與修正**
```
你：執行結果是「找不到標籤類型」，請幫我修正
```

> 「這個循環可能要重複好幾次。沒關係，AI 會幫你改。」

---

## Slide 5: 決定產出類型

**標題**：domain（長期）vs 一次性

---

### 📝 講師備忘錄

**分類決策（5 分鐘）**：

| 判斷標準 | domain（長期） | 一次性腳本 |
|:--------|:--------------|:---------|
| 使用頻率 | 常常需要 | 只用一次 |
| 通用性 | 多專案適用 | 特定專案 |
| 複雜度 | 需要標準化 | 簡單直接 |
| 存放位置 | `domain/*.md` | `scratch/*.js` |

---

**提問互動**：

> 「如果你開發了一個『自動標記消防栓』的工具：
> - 這個功能會不會常常用到？（會 → domain）
> - 只有這個專案需要嗎？（是 → 一次性）」

---

## 📋 本堂課總結

**記住這個流程**：
1. 先評估現有工具能不能完成
2. 不能的話，描述需求給 AI
3. 請 AI 查 Revit SDK
4. 請 AI 設計規格和產生程式碼
5. 測試 → 修正 → 直到成功
6. 決定是 domain 還是一次性

**核心觀念**：
> 「你是需求的提出者，AI 是程式碼的撰寫者。
> 你們是合作關係，不是你要學會 AI 的語言。」
